> 이 화면은 휴관 일정의 기간·제목을 수정하는 화면이 아니다. 휴관일 관리 캘린더에서 선택한 날짜의 영업 시작·종료 시간을 수정한다.

## 목적

토이빌리지 관리자가 특정 날짜의 영업 시작·종료 시간을 확인하고 수정한다. 저장된 날짜별 값이 없으면 기본 영업시간을 제안하되, 저장 전까지는 서버 데이터로 확정하지 않는다.

## 라우트와 진입

- 휴관일 관리 캘린더의 날짜 클릭 → `/notices/guide/hours/:date`로 이동한다.
- `date`는 로컬 date-only 형식 `YYYY-MM-DD`다.
- 화면 제목은 `date`를 기준으로 `M월 D일 영업시간` 형식으로 표시한다.
- `뒤로가기` 클릭 → 변경 내용을 저장하지 않고 `/notices/guide`로 이동한다.
- 저장 성공 → `/notices/guide`로 이동한다.
- 존재하지 않는 날짜를 포함한 잘못된 `date` 경로 → `/notices/guide`로 replace 이동한다.

## 초기 데이터와 표시

- 페이지 진입 시 선택 날짜의 영업시간을 조회한다.
- 저장값이 있으면 저장된 `opensAt`, `closesAt`을 표시한다.
- 저장값이 없으면 다음 기본값을 표시한다.
  - 영업 시작: `오전 07:40`
  - 영업 종료: `오후 07:40`
- API 값은 24시간제 `HH:mm`, 화면 입력은 12시간제 `오전/오후 + hh:mm`으로 변환한다.
- `00:xx`는 `오전 12:xx`, `12:xx`는 `오후 12:xx`로 표시한다.
- 조회가 끝나기 전에는 기본값을 저장값인 것처럼 확정해 표시하지 않는다.
- Figma에 로딩·조회 실패 화면이 없으므로 새로운 spinner, skeleton 또는 임의 오류 문구는 이 spec에서 확정하지 않는다.

## 시간 입력

### 시

- `영업 시작 시`, `영업 종료 시`는 숫자 1–12만 유효하다.
- 최대 두 자리까지 입력할 수 있다.
- 숫자가 아닌 문자는 입력값에서 제외한다.
- 한 자리 값은 포커스를 잃을 때 앞에 `0`을 붙여 두 자리로 표시한다.

### 분

- `영업 시작 분`, `영업 종료 분`은 숫자 0–59만 유효하다.
- 최대 두 자리까지 입력할 수 있다.
- 숫자가 아닌 문자는 입력값에서 제외한다.
- 한 자리 값은 포커스를 잃을 때 앞에 `0`을 붙여 두 자리로 표시한다.

### 오전·오후

- 각 카드의 `오전`, `오후` 중 하나만 선택할 수 있다.
- 선택 변경은 같은 카드의 시·분 값을 보존한다.
- 오전·오후 버튼에서 Enter 또는 Space를 누르면 해당 선택만 변경하며 폼을 제출하지 않는다.

## 검증

`저장하기` 클릭 또는 시·분 입력에서 Enter를 누르면 다음 순서로 검증한다.

1. 시·분 형식과 범위가 유효하지 않음 → 저장하지 않고 `시간을 확인해 주세요`를 표시한다.
2. 24시간제로 변환한 종료 시간이 시작 시간과 같거나 빠름 → 저장하지 않고 `영업 종료 시간은 시작 시간보다 늦어야 합니다`를 표시한다.
3. 모든 값이 유효함 → 오류 메시지를 지우고 저장 요청을 한 번 전송한다.

- 종료 시간이 다음 날로 넘어가는 영업은 이번 범위에서 지원하지 않는다.
- 검증 실패 시 입력값과 현재 페이지를 유지한다.
- 사용자가 값을 다시 편집할 때 기존 오류를 즉시 지울지는 Figma에 정의되어 있지 않다. 다음 제출 시 재검증하는 동작만 보장한다.

## 저장

- 유효한 값은 선택 날짜와 함께 24시간제 `HH:mm`으로 저장한다.
- 저장 중에는 `저장하기`를 비활성화하고 라벨을 `저장 중`으로 바꿔 중복 제출을 막는다.
- 저장 성공 → 선택 날짜의 영업시간 query를 갱신하고 `/notices/guide`로 이동한다.
- 저장 실패 → 현재 입력값과 페이지를 유지하고 `저장하지 못했습니다. 다시 시도해 주세요.`를 표시한다.
- 실패 후 사용자는 같은 화면에서 다시 저장할 수 있다.
- 성공 toast는 Figma에 없으므로 추가하지 않는다. 목록 이동과 재진입 시 저장값 확인이 성공 피드백이다.

## 데이터와 API 경계

```ts
interface OperatingHours {
  date: string
  opensAt: string
  closesAt: string
}

type UpdateOperatingHoursRequest = OperatingHours
```

- `date`: `YYYY-MM-DD`
- `opensAt`, `closesAt`: 24시간제 `HH:mm`
- 조회 endpoint 후보: `GET /operating-hours/:date`
- 수정 endpoint 후보: `PUT /operating-hours/:date`
- 실제 API 계약 전에는 `entities/operating-hours`의 localStorage 목업 어댑터를 사용한다.
- 서버 상태와 mutation은 TanStack Query, HTTP 연동 시 `src/shared/api/axios.ts`를 사용한다.
- 시·분·오전/오후 입력과 검증 메시지는 feature 로컬 상태로 관리한다.
- query key 후보: `['operating-hours', date]`

API 경로, 응답 타입과 기본 영업시간의 서버 소유 여부는 백엔드 계약 전까지 후보로 취급한다.

## 접근성

- `뒤로가기`는 실제 링크이며 접근 가능한 이름 `뒤로가기`를 제공한다.
- 화면 제목은 `h1`이다.
- 각 시간 카드는 `영업 시작`, `영업 종료` 이름을 가진 group으로 노출한다.
- 시·분 입력은 각각 `영업 시작 시`, `영업 시작 분`, `영업 종료 시`, `영업 종료 분`의 접근 가능한 이름을 제공한다.
- 오전·오후 선택은 button과 `aria-pressed`로 현재 상태를 전달한다.
- 포커스 상태는 색만이 아닌 명확한 outline으로 표시한다.
- 오류·저장 실패 메시지는 보조기기에 전달되는 status 영역으로 표시한다.
- 키보드 순서는 `뒤로가기` → 영업 시작 시 → 영업 시작 분 → 영업 시작 오전/오후 → 영업 종료 시 → 영업 종료 분 → 영업 종료 오전/오후 → `저장하기`다.
- 시·분 입력에서 Enter → `저장하기`와 같은 검증·저장 흐름을 실행한다.
- IME 조합 중 Enter는 제출하지 않는다.

## 반응형

- Figma에는 1920×1080 데스크톱만 정의되어 있다.
- 980px 이하에서는 영업 시작·종료 카드를 한 열로 배치한다.
- 각 카드와 시간 입력은 가용 너비 안에서 줄어들되 잘리거나 가로 스크롤을 만들지 않는다.
- 모바일 정밀 배치는 Figma 미제공 항목이며 데스크톱 수용 기준을 대체하지 않는다.

## 기능 테스트 수용 기준

- S1: 휴관일 관리 캘린더의 날짜 클릭 → `/notices/guide/hours/:date`로 이동하고 `M월 D일 영업시간` 제목이 보인다.
- S2: 저장값이 없는 날짜 진입 → 영업 시작 `오전 07:40`, 영업 종료 `오후 07:40`이 보인다.
- S3: 유효한 영업시간으로 `저장하기` 클릭 → 저장 요청이 한 번 실행되고 `/notices/guide`로 이동한다.
- S4: 저장한 날짜에 다시 진입 → 저장한 시·분·오전/오후 값이 보인다.
- S5: 값을 수정한 뒤 `뒤로가기` 클릭 → 저장하지 않고 `/notices/guide`로 이동한다.
- S6: 시·분의 형식 또는 범위가 잘못됨 → 요청하지 않고 `시간을 확인해 주세요`가 보인다.
- S7: 종료 시간이 시작 시간과 같거나 빠름 → 요청하지 않고 `영업 종료 시간은 시작 시간보다 늦어야 합니다`가 보인다.
- S8: 저장 실패 → 입력값과 현재 URL이 유지되고 다시 저장할 수 있다.
- S9: 시·분 입력에서 Enter → 유효하면 저장하고, 유효하지 않으면 동일한 오류를 표시한다.
- S10: 잘못된 날짜 URL 진입 → `/notices/guide`로 replace 이동한다.
- S11: 저장 중 저장 버튼을 반복 클릭 → 저장 요청은 중복 전송되지 않는다.
- S12: 키보드만 사용 → 모든 입력·오전/오후 선택·저장·뒤로가기를 수행할 수 있다.
