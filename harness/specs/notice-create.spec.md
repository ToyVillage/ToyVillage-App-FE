Figma `1541:1442`에서 검증된 생성 진입 CTA의 문구는 `공지 생성하기`이고 목적지는 `/notices/list/create`다. 생성 화면 자체의 node ID와 오류·성공 상태는 아직 로컬 근거에서 확인되지 않았다. 아래 입력 구조와 요청 계약은 구현 가능한 초안을 위한 제품 가정이며, 생성 화면 node가 확인되면 시각 배치와 문구를 우선 대조한다.

## 목적

토이빌리지 운영 관리자가 공지 분류, 제목과 내용을 작성해 새로운 공지를 게시한다. 입력 오류나 요청 실패가 발생해도 작성 내용을 잃지 않고 다시 시도할 수 있어야 한다.

## 범위

- 포함: 공지 분류 선택, 제목·내용 입력, 파일 첨부 UI, 필수값 검증, 생성 요청, 목록 복귀
- 제외: 임시 저장, 예약 게시, 공개 범위, 수정·삭제, 리치 텍스트 편집, 미리보기

## 라우트와 진입

- `/notices/list`의 `공지 생성하기` 클릭 → `/notices/list/create`로 이동한다.
- `/notices/list/create` → 공지 생성 페이지를 표시한다.
- 생성 페이지의 `뒤로가기` 클릭 → `/notices/list`로 이동한다.
- 브라우저 새로고침 후에도 생성 페이지 라우트가 직접 열려야 한다.

## 화면 구조

데스크톱의 공통 콘텐츠 영역과 전역 메뉴 위치는 Figma `1541:1442` 및 `DESIGN.md`를 따른다.

1. 좌측 상단 전역 메뉴 버튼
2. 콘텐츠 상단 `뒤로가기`
3. 공지 입력 폼
   - `제목 *`
   - `분류`
   - `내용 *`
   - `첨부파일`
4. 주요 액션 `생성하기`

생성 화면 고유의 카드 크기, 필드 간격, 입력 높이와 버튼 위치는 node ID 확인 전 확정하지 않는다. 목록 화면에서 관찰된 배경 `#F5F5F7`, 흰색 surface, 검은색 주요 액션과 Wanted Sans는 기존 theme 및 컴포넌트 규칙으로 재사용한다.

## 입력 필드

### 분류

- label: `분류`
- 단일 선택 필드이며 `전체`를 기본값으로 사용한다.
- 초기값은 `전체`다.
- 옵션은 서버 또는 공지 분류 query에서 제공한다.
- 선택한 팀 분류를 삭제하면 `전체`로 돌아간다.

### 제목

- label: `제목 *`
- 필수 한 줄 텍스트 필드다.
- placeholder 초안: `제목을 입력해주세요`
- 제출 시 앞뒤 공백을 제거한다.
- 공백 제거 후 빈 문자열이면 입력하지 않은 것으로 판단한다.
- Figma와 API에 최대 길이가 확인되지 않았으므로 임의의 글자 수 제한이나 카운터를 추가하지 않는다.

### 내용

- label: `내용 *`
- 필수 여러 줄 일반 텍스트 필드다.
- placeholder 초안: `내용을 입력해주세요`
- 줄바꿈은 유지하고, 값 전체의 앞뒤 공백만 제거한다.
- 공백 제거 후 빈 문자열이면 입력하지 않은 것으로 판단한다.
- Figma에서 편집 도구가 확인되지 않았으므로 `contenteditable` 또는 리치 텍스트 에디터를 도입하지 않는다.

### 첨부파일

- label 또는 group name: `첨부파일`
- 첨부파일은 선택 사항이다.
- 첨부파일 영역은 항상 화면에 노출한다.
- 첨부파일 영역에는 첨부 상태 카드와 업로드 dropzone을 함께 표시한다.
- 첨부 상태 카드는 upload dropzone 위에 배치한다.
- 첨부된 파일이 없을 때도 참조 이미지처럼 내용이 없는 빈 첨부 상태 카드를 표시한다.
- upload dropzone은 클릭과 drag-and-drop으로 파일을 추가할 수 있어야 한다.
- upload control은 keyboard focus를 받을 수 있어야 하며 접근 가능한 이름은 `파일 업로드`다.
- 파일 선택 dialog에서는 여러 파일을 한 번에 선택할 수 있어야 한다.
- 첨부된 파일은 첨부 상태 카드 안에서 파일명 chip으로 표시한다.
- 각 파일명 chip은 해당 파일을 제거하는 control을 제공한다.
- 제거 control의 접근 가능한 이름은 `${파일명} 삭제` 형식을 사용한다.
- 첨부 파일을 모두 제거하면 첨부 상태 카드는 다시 내용이 없는 빈 상태로 돌아간다.
- 개별 파일 크기는 최대 50MB까지 허용한다.
- 50MB를 초과한 파일은 첨부하지 않고 오류 상태로 알려야 한다. 정확한 오류 문구와 시각 상태는 제품·디자인 확인 전 확정하지 않는다.

## 동작

### 초기 상태

- 분류는 `전체`가 선택되고 제목과 내용은 비어 있다.

### 분류 팀 관리

- 팀 분류를 선택하거나 포커스해도 핑크 배경을 적용하지 않는다.
- 각 팀 분류의 `X`는 실제 삭제 control이며, 누르면 해당 팀 pill이 화면에서 사라진다.
- 현재 선택된 팀을 삭제하면 분류는 `전체`로 돌아간다.
- `+ 팀 추가`를 누르면 `팀 추가하기` 모달을 열며 팀 이름 입력창을 포커스해도 윤곽선을 표시하지 않는다.
- 모달의 `취소` 또는 Escape는 팀을 추가하지 않고 닫으며 `+ 팀 추가`로 포커스를 복귀한다.
- 팀 이름을 입력하고 `다음`을 누르면 해당 이름의 팀 pill을 추가하고 선택한다.
- `생성하기`는 화면에 항상 노출한다.
- 필수값 검증은 `생성하기`를 눌렀을 때 수행한다.
- 생성 요청 전에는 서버 상태나 성공 메시지를 표시하지 않는다.

### 뒤로가기와 작성 중 이탈

- 입력값이 없는 상태에서 `뒤로가기` 클릭 → `/notices/list`로 이동한다.
- 제목·내용·분류·팀 목록·첨부파일 중 하나라도 초기 상태에서 바뀐 뒤 화면의 `뒤로가기`, 사이드바 링크 또는 브라우저 뒤로가기로 생성 페이지를 벗어나려 하면 이탈 확인 모달을 표시한다.
- 모달 제목은 `정말 나가시겠습니까?`, 설명은 `저장하지 않고 돌아갈 시\n입력된 정보가 삭제됩니다`다.
- `취소` 또는 Escape → 모달을 닫고 입력값과 현재 URL을 유지한다.
- `확인` → 시도했던 경로로 이동한다.
- 생성 성공에 따른 `/notices/list` 이동은 이탈 확인 대상에서 제외한다.
- 새로고침·탭 닫기처럼 브라우저 문서 자체를 벗어나는 동작은 브라우저 기본 이탈 경고로 보호한다.

### 이탈 확인 모달

- 전체 화면에 검은색 50% overlay를 표시한다.
- 중앙 모달은 600×300px, 흰색 배경, 20px radius다.
- 제목은 28px, 설명은 회색 20px로 중앙 정렬한다.
- 하단 `취소`, `확인` 버튼은 각각 100×48px이며 사이 간격은 16px다.
- `취소`는 흰 배경·회색 border/text, `확인`은 검은 배경·흰 텍스트다.
- 모달은 `role="alertdialog"`, `aria-modal="true"`와 제목·설명 연결을 제공한다.
- 모달이 열리면 안전한 기본 행동인 `취소`로 포커스를 이동하고 Tab 포커스를 두 버튼 안에 유지한다.
- 모달이 열린 동안 배경 콘텐츠를 키보드와 스크린 리더 탐색에서 제외하며 바깥 영역 클릭으로 닫지 않는다.

### 생성 검증

`생성하기` 클릭 시 제목 → 내용 순서로 검증한다.

- 제목이 비어 있음 → 요청하지 않고 `제목을 입력해 주세요`를 알린다.
- 내용이 비어 있음 → 요청하지 않고 `내용을 입력해 주세요`를 알린다.
- 오류는 한 번에 하나씩 검증 순서에 따라 모달로 표시한다.
- 모달의 `확인` 또는 Escape → 모달을 닫고 해당 오류 입력으로 포커스를 이동한다.
- 사용자가 값을 수정하고 다시 제출하면 다음 오류를 검증한다.

### 검증 모달

- 전체 화면에 검은색 50% overlay를 표시한다.
- 중앙 모달은 560×320px, 흰색 배경, 20px radius다.
- 메시지는 상단 중앙에 28px SemiBold로 표시한다.
- 하단 `확인` 버튼은 모달 내부 너비 520px, 검은색 배경, 흰색 28px Medium 텍스트다.
- 모달은 `role="alertdialog"`, `aria-modal="true"`와 메시지 연결을 제공한다.
- 모달이 열리면 `확인`으로 포커스를 이동하고 Tab 포커스를 모달 안에 유지한다.
- 모달이 열린 동안 배경 콘텐츠를 키보드와 스크린 리더 탐색에서 제외한다.
- 바깥 영역 클릭으로는 닫지 않는다.

### 생성 요청

- 모든 값이 유효함 → 생성 요청을 한 번 전송한다.
- 요청 값의 제목과 내용은 정규화한 값을 사용한다.
- 요청 중에는 중복 제출을 막기 위해 `생성하기`를 비활성화하고 진행 중임을 텍스트로 알린다.
- 성공 → 공지 목록 query를 갱신하고 `/notices/list`로 이동한다.
- 실패 → 현재 페이지, 선택한 분류, 제목과 내용을 유지해 다시 시도할 수 있게 한다.
- 서버가 필드 오류를 반환하면 해당 필드와 연결하고, 일반 오류면 폼 상단 상태 영역에 표시한다.
- 성공·실패 피드백의 정확한 문구와 시각 상태는 Figma 또는 제품 정책 확인 전 확정하지 않는다.

## 데이터와 API 경계

API 확정 전 후보 계약은 다음과 같다.

```ts
interface CreateNoticeRequest {
  category: string
  title: string
  content: string
  attachments?: File[]
}

interface Notice {
  id: string
  category: string
  title: string
  content: string
  createdAt: string
}
```

- 후보 endpoint: `POST /notices`
- 목록 query key 후보: `['notices']`
- 서버 상태와 mutation은 TanStack Query, HTTP는 `src/shared/api/axios.ts`를 사용한다.
- 폼 값, touched와 클라이언트 검증 오류는 feature 로컬 상태로 관리한다.
- 현재 `Notice`의 `date` 표시값은 추후 `createdAt`에서 포맷하는 방향을 우선 검토한다.
- mock 저장소가 파일 본문을 직렬화하지 못하는 동안에는 생성 성공 시 첨부 파일명을 목록에 저장하지 않아도 된다. 실제 API 연결 시 multipart 또는 사전 업로드 계약을 확정한다.
- API endpoint, 분류 식별자 타입, 요청·응답 필드와 날짜 형식은 백엔드 계약 전 확정해야 한다.

## 컴포넌트 구조와 소유권

- `CreateNoticePage` (`pages/notices/notice`) — 페이지 레이아웃, 뒤로가기와 생성 폼 조합
- `NoticeForm` (`features/create-notice`) — 입력 상태, 검증과 생성 mutation
- `NoticeCategoryField` (`features/create-notice` 내부) — 분류 옵션 조회와 선택
- `NoticeAttachmentField` (`features/create-notice` 내부) — 빈 첨부 상태, dropzone, 다중 파일 선택, chip 표시와 제거
- `CreateNoticeRequest`, 생성 결과 타입 (`entities/notice`) — 공지 도메인 모델
- `ValidationDialog` (`shared/ui`) — 공지와 휴관일 생성에서 함께 사용하는 Figma 검증 모달
- 공통 `FormField`, `Textarea`, 오류 요약이 이미 존재하면 재사용한다. 구조가 이 화면에만 필요하면 feature 내부에 둔다.
- 새 form, select 또는 rich-text 의존성은 추가하지 않는다.

## 접근성

- `뒤로가기`는 실제 링크 또는 버튼으로 구현하고 접근 가능한 이름을 제공한다.
- 모든 label은 대응하는 control과 프로그램적으로 연결한다.
- 필수 상태는 별표뿐 아니라 `required` 또는 `aria-required="true"`로 전달한다.
- 첨부파일 영역은 `role="group"` 또는 동등한 구조로 `첨부파일` 이름을 제공한다.
- upload control은 Tab 순서에 포함하고 `파일 업로드` 이름으로 조작할 수 있어야 한다.
- 각 첨부 chip의 제거 control은 `${파일명} 삭제` 이름을 제공한다.
- 검증 모달을 닫으면 오류가 발생한 control로 포커스를 이동한다.
- 기본 키보드 순서는 `뒤로가기` → 제목 → 분류 → 내용 → `파일 업로드` → `생성하기`다.
- 첨부된 파일이 있으면 파일 chip 제거 control도 DOM 배치에 맞게 Tab 순서에 포함한다.
- select와 textarea를 포함한 모든 조작은 키보드만으로 완료할 수 있어야 한다.
- focus-visible 상태는 색만이 아닌 명확한 outline으로 표현한다.
- 요청 상태 변화와 일반 서버 오류는 적절한 live region으로 알린다.

## 반응형

- 1920×1080 목록 프레임의 콘텐츠 정렬을 데스크톱 기준으로 사용한다.
- 980px 이하에서는 폼과 액션을 한 열로 배치하고 각 control은 가용 너비를 사용한다.
- 좁은 화면에서도 내용 입력과 검증 모달이 잘리거나 가로 스크롤을 만들지 않아야 한다.
- 모바일 정밀 배치는 Figma 미제공 항목이며 데스크톱 수용 기준을 대체하지 않는다.

## 기능 테스트 수용 기준

- S1: `/notices/list`에서 `공지 생성하기` 클릭 → `/notices/list/create`로 이동한다.
- S2: 생성 페이지 진입 → 별도 페이지 제목 없이 `뒤로가기`, 제목, 분류, 내용, 빈 첨부 카드, upload dropzone과 `생성하기`가 보인다.
- S3: `뒤로가기` 클릭 → `/notices/list`로 이동한다.
- S4: 빈 폼 제출 → 제목 오류 모달을 표시하고, 확인 후 제목으로 포커스가 이동한다.
- S5: 제목만 입력하고 제출 → 내용 오류 모달을 표시하고, 확인 후 내용으로 포커스가 이동한다.
- S6: 제목이나 내용이 공백뿐인 상태로 제출 → 제목과 내용 오류를 화면 순서대로 모달로 검증한다.
- S7: 유효한 값으로 제출 성공 → 요청이 한 번 전송되고 `/notices/list`로 이동한다.
- S8: 요청 중 `생성하기` 재클릭 → 중복 요청을 전송하지 않는다.
- S9: 생성 실패 → 분류, 제목과 내용이 유지되고 다시 제출할 수 있다.
- S10: 키보드만 사용 → 모든 입력, upload control, 오류 확인, 제출과 뒤로가기를 수행할 수 있다.
- S11: 생성 페이지 진입 → 첨부파일 영역에 빈 첨부 카드가 upload dropzone 위에 보인다.
- S12: 여러 파일을 한 번에 선택 → 각 파일명을 chip으로 표시하고 각 chip에 제거 control이 보인다.
- S13: 첨부 파일 제거 → 파일 chip이 사라지고 빈 첨부 카드 상태로 돌아간다.
- S14: 50MB 초과 파일 선택 → 파일을 첨부하지 않고 오류 상태를 알린다.
- S15: 작성 후 `뒤로가기` 클릭 → Figma `2431:2235` 이탈 확인 모달을 표시한다.
- S16: 이탈 확인 모달에서 `취소` 또는 Escape → 입력값과 현재 URL을 유지하고 이탈을 시도한 control로 포커스를 복귀한다.
- S17: 이탈 확인 모달에서 `확인` → 시도했던 경로로 이동한다.
- S18: 작성 중 사이드바 링크 또는 브라우저 뒤로가기 → 동일한 이탈 확인 모달을 표시한다.
- S19: 생성 성공 → 이탈 확인 없이 `/notices/list`로 이동한다.
